\section{Autotools basics}

=== {{\tt configure.ac} language}
  \begin{itemize}
  \item Really a shell script
  \item Processed through the ``` m4 ``` preprocessor
  \item Shell script augmented with special constructs for portability:
    \begin{itemize}
    \item ``` AS_IF ``` instead of shell ``` if ... then .. fi ```
    \item ``` AS_CASE ``` instead of shell ``` case ... esac ```
    \item etc.
    \end{itemize}
  \item {\em autoconf} provides a large set of {\em m4} macros to
    perform most of the usual tests
  \item Make sure to quote macro arguments with ``` [] ```
  \end{itemize}


=== [fragile]{Minimal {\tt configure.ac}}
  \begin{block}{configure.ac}
  \begin{minted}{bash}
AC_INIT([hello], [1.0])
AC_OUTPUT
  \end{minted}
  \end{block}
  \begin{itemize}
  \item ``` AC_INIT ```
    \begin{itemize}
    \item Every configure script must call ``` AC_INIT ``` before doing
      anything else that produces output.
    \item Process any command-line arguments and perform
      initialization and verification.
    \item Prototype:\\
      ``` AC_INIT (package, version, [bug-report], [tarname], [url]) ```
    \end{itemize}
  \item ``` AC_OUTPUT ```
    \begin{itemize}
    \item Every ``` configure.ac ```, should finish by calling
      ``` AC_OUTPUT ```.
    \item Generates and runs config.status, which in turn creates the
      makefiles and any other files resulting from configuration.
    \end{itemize}
  \end{itemize}


=== [fragile]{Minimal {\tt configure.ac} example}

  \begin{block}{}
    \begin{minted}[fontsize=\scriptsize]{console}
$ cat configure.ac AC_INIT([hello], [1.0])
AC_OUTPUT
$ ls configure.ac
$ autoreconf -i
$ ls autom4te.cache  configure  configure.ac
$ ./configure configure: creating ./config.status
$ ls autom4te.cache  config.log    config.status configure       configure.ac
$ wc -l configure
2390 configure
\end{minted}
\end{block}



=== {Additional basic macros}
  \begin{itemize}
  \item ``` AC_PREREQ ```
    \begin{itemize}
    \item Verifies that a recent enough version of {\em autoconf} is used
    \item ``` AC_PREREQ([2.68]) ```
    \end{itemize}
  \item ``` AC_CONFIG_SRCDIR ```
    \begin{itemize}
    \item Gives the path to one source file in your project
    \item Allows {\em autoconf} to check that it is really where it
      should be
    \item ``` AC_CONFIG_SRCDIR([hello.c]) ```
    \end{itemize}
  \item ``` AC_CONFIG_AUX_DIR ```
    \begin{itemize}
    \item Tells {\em autoconf} to put the auxiliary build tools it
      requires in a different directory, rather than the one of
      ``` configure.ac ```
    \item Useful to keep cleaner build directory
    \end{itemize}
  \end{itemize}


=== {Checking for basic programs}
  \begin{itemize}
  \item ``` AC_PROG_CC ```, makes sure a C compiler is available
  \item ``` AC_PROG_CXX ```, makes sure a C++ compiler is available
  \item ``` AC_PROG_AWK ```, ``` AC_PROG_GREP ```, ``` AC_PROG_LEX ```,
    ``` AC_PROG_YACC ```, etc.
  \end{itemize}

=== {{\tt AC\_CONFIG\_FILES}}
  \begin{itemize}
  \item ``` AC_CONFIG_FILES (file..., [cmds], [init-cmds]) ```
  \item Make ``` AC_OUTPUT ``` create each file by copying an input
    ``` file ``` (by default ``` file.in ```), substituting the {\em
      output variable values}.
  \item Typically used to turn the Makefile templates
    ``` Makefile.in ``` files into final ``` Makefile ```.
  \item Example:\\
    ``` AC_CONFIG_FILES([Makefile src/Makefile]) ```
  \item ``` cmds ``` and ``` init-cmds ``` are rarely used, see the {\em
      autoconf} documentation for details.
  \end{itemize}


=== [fragile]{Output variables}
  \begin{itemize}
  \item {\em autoconf} will replace ``` @variable@ ``` constructs by
    the appropriate values in files listed in ``` AC_CONFIG_FILES ```
  \item Long list of standard variables replaced by {\em autoconf}
  \item Additional shell variables declared in ``` configure.ac ``` can
    be replaced using ``` AC_SUBST ```
  \item The following three examples are equivalent:
    \begin{block}{}
\begin{verbatim}
AC_SUBST([FOO], [42])
\end{verbatim}
    \end{block}

    \begin{block}{}
\begin{verbatim}
FOO=42
AC_SUBST([FOO])
\end{verbatim}
    \end{block}

    \begin{block}{}
\begin{verbatim}
AC_SUBST([FOO])
FOO=42
\end{verbatim}
    \end{block}

  \end{itemize}

  constructs
\end{itemize}

=== {Writing {\tt Makefile.in}?}
  \begin{itemize}
  \item At this point, we have seen the very basics of {\em autoconf}
    to perform the configuration side of our software
  \item We could use ``` AC_CONFIG_FILES ``` to generate
    ``` Makefile ``` from ``` Makefile.in ```
  \item However, writing a ``` Makefile.in ``` properly is not easy,
    especially if you want to:
    \begin{itemize}
    \item be portable
    \item automatically handle dependencies
    \item support conditional compilation
    \end{itemize}
  \item For these reasons, ``` Makefile.in ``` are typically not
    written manually, but generated by {\em automake} from a
    ``` Makefile.am ``` file
  \end{itemize}


=== {{\tt Makefile.am} language}
  \begin{itemize}
  \item Really just a ``` Makefile ```
    \begin{itemize}
    \item You can include regular {\em make} code
    \end{itemize}
  \item Augmented with {\em automake} specific constructs that are
    expanded into regular {\em make} code
  \item For most situations, the {\em automake} constructs are
    sufficient to express what needs to be built
  \end{itemize}


=== [fragile]{{\tt Makefile.am} minimal example}

  \begin{itemize}
  \item The minimal example of ``` Makefile.am ``` to build just one C
    file into a program is only two lines:
  \end{itemize}

  \begin{block}{Makefile.am}
\begin{verbatim}
bin_PROGRAMS = hello hello_SOURCES = main.c
\end{verbatim}
  \end{block}

  \begin{itemize}
  \item Will compile ``` main.c ``` to ``` main.o ```
  \item And link ``` hello.o ``` into the ``` hello ``` executable
  \item Which will be installed in ``` $prefix/bin ```
  \end{itemize}


=== [fragile]{Enabling {\em automake} in {\tt configure.ac}}
  \begin{itemize}
  \item To enable {\em automake} usage in ``` configure.ac ```, you
    need:
    \begin{itemize}
    \item A call to ``` AM_INIT_AUTOMAKE ```
    \item Generate the ``` Makefile ``` using ``` AC_CONFIG_FILES ```
    \end{itemize}
  \item {\em automake} will generate the ``` Makefile.in ``` at {\em
      autoreconf} time, and {\em configure} will generate the final
    ``` Makefile ```
  \end{itemize}

  \begin{block}{configure.ac}
\begin{verbatim}
AC_INIT([hello], [1.0])
AM_INIT_AUTOMAKE([foreign 1.13])
AC_PROG_CC
AC_CONFIG_FILES([Makefile])
AC_OUTPUT
\end{verbatim}
  \end{block}



=== {{\tt AM\_INIT\_AUTOMAKE}}

  \begin{itemize}
  \item ``` AM_INIT_AUTOMAKE([OPTIONS]) ```
  \item Interesting options:
    \begin{itemize}
    \item ``` foreign ```, tells {\em automake} to not require all the
      GNU Coding Style files such as ``` NEWS ```, ``` README ```,
      ``` AUTHORS ```, etc.
    \item ``` dist-bzip2 ```, ``` dist-xz ```, etc. tell {\em automake}
      which tarball format should be generated by ``` make dist ```
    \item ``` subdir-objects ``` tells {\em automake} that the objects
      are placed into the subdirectory of the build directory
      corresponding to the subdirectory of the source file
    \item {\em version}, e.g ``` 1.14.1 ```, tells the minimal
      ``` automake ``` version that is expected
    \end{itemize}
  \end{itemize}


=== [fragile]{{\tt Makefile.am} syntax}
  \begin{itemize}
  \item An {\em automake} parsable Makefile.am is composed of {\bf
      product list variables}:
    \begin{block}{}
\begin{verbatim}
bin_PROGRAMS = hello
\end{verbatim}
\end{block}
  \item And {\bf product source variables}:
    \begin{block}{}
\begin{verbatim}
hello_SOURCES = main.c
\end{verbatim}
\end{block}

\end{itemize}


=== [fragile]{Product list variables}

    \begin{block}{}
{\small
\begin{verbatim}
[modifier-list]prefix_PRIMARY = product1 product2 ...
\end{verbatim}}
\end{block}

  \begin{itemize}

\item ``` prefix ``` is the installation prefix, i.e. where it should be
  installed
  \begin{itemize}
  \item All ``` *dir ``` variables from {\em autoconf} can be used,
    without their ``` dir ``` suffix: use ``` bin ``` for ``` bindir ```
  \item E.g.: ``` bindir ```, ``` libdir ```, ``` includedir ```,
    ``` datadir ```, etc.
  \end{itemize}

\item ``` PRIMARY ``` describes what type of thing should be built:
  \begin{itemize}
  \item ``` PROGRAMS ```, for executables
  \item ``` LIBRARIES ```, ``` LTLIBRARIES ```, for libraries
  \item ``` HEADERS ```, for publicly installed header files
  \item ``` DATA ```, arbitrary data files
  \item ``` PYTHON ```, ``` JAVA ```, ``` SCRIPTS ```
  \item ``` MANS ```, ``` TEXINFOS ```, for documentation
  \end{itemize}

\item After the ``` = ``` sign, list of products to be generated

\end{itemize}



=== [fragile]{Product source variables}

    \begin{block}{}
{\small
\begin{verbatim}
[modifier-list]product_SOURCES = file1 file2 ...
\end{verbatim}}
\end{block}

\begin{itemize}

\item The ``` product ``` is the normalized name of the product, as
  listed in a {\em product list variable}
  \begin{itemize}
  \item The normalization consists in replacing special characters
    such as ``` . ``` or ``` + ``` by ``` _ ```. For example,
    ``` libfoo+.a ``` in a {\em product list variable} gives the
    ``` libfoo__a_SOURCES ``` product source variable.
  \end{itemize}
\item ``` _SOURCES ``` is always used, it's not like a configurable {\em
    primary}.
  \begin{itemize}
  \item Contains the list of files containing the source code for the
    product to be built.
  \item Both source files {\em and} header files should be listed.
  \end{itemize}
\end{itemize}



=== [fragile]{Example: building multiple programs}
  \begin{block}{Makefile.am}
    {\small
\begin{verbatim}
bin_PROGRAMS = hello test hello_SOURCES = main.c common.c common.h test_SOURCES = test.c common.c common.h
\end{verbatim}}
  \end{block}
  \begin{itemize}
  \item Building two programs: ``` hello ``` and ``` test ```
  \item Shared source files: ``` common.c ``` and ``` common.h ```
  \end{itemize}
